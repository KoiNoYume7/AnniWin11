# Architecture

Technical architecture overview for AnniWin11.

For the long-term design vision, see [VISION.md](VISION.md).

---

## Overview

AnniWin11 is a PowerShell 7+ automation suite for Windows 11 post-install setup. It follows a two-phase model per application: **install** (via winget/internet/local/terminal) then **config restore** (raw file copy from backup store).

The project is evolving from a personal setup script into a general-purpose public tool with automated app detection, config discovery, and a community-maintained lookup table. This document covers both the current architecture (v0.1.x) and planned additions.

## Entry Point

```
setup.bat
  -> UAC elevation check
  -> PowerShell 7 bootstrap (install if missing)
  -> Launch src/Main.ps1 via pwsh
```

## Directory Layout

```
AnniWin11/
  setup.bat              Entry point (batch, handles elevation)
  src/                   User-facing scripts (orchestration, features)
  lib/                   Reusable modules and helpers (project-agnostic where possible)
  config/                Configuration files (examples committed, generated files gitignored)
  docs/                  Documentation (architecture, vision)
  logs/                  Runtime log files (gitignored)
```

## Module Architecture

### lib/AnniLog.psm1

Reusable logging module. No AnniWin11-specific logic.

- **State**: module-scoped hashtable (`$script:AnniLogState`)
- **Levels**: ERROR (0), WARNING (1), INFO (2), SUCCESS (2), DEBUG (3)
- **Output**: coloured console via `Write-Host` + timestamped append to log file
- **Lifecycle**: `Initialize-AnniLog` -> `Write-AnniLog` (repeated) -> `Close-AnniLog`
- **Stopwatch**: optional, started on init, elapsed time logged on close

### lib/AnniLogo.psm1

Reusable ASCII art renderer. No AnniWin11-specific content.

- **Input**: array of strings (`$AsciiLines`), start/end colour hashtables
- **Rendering**: per-character ANSI 24-bit colour escape sequences
- **Gradient**: horizontal linear interpolation between two RGB colours

### lib/Config.ps1

Dot-sourced helper (not a module). Provides:

- **`Get-ProjectRoot`** -- resolves repo root from `$PSScriptRoot`
- **`Get-ConfigPath`** / **`Get-LogPath`** -- resolve paths relative to project root
- **`Read-JsoncFile`** -- strips JSONC comments (block + line + inline) then parses
- **`Read-JsonFile`** -- plain JSON reader
- **`Resolve-BackupPath`** -- expands path tokens (`appdata`, `localappdata`, `programdata`, `absolute`) to full filesystem paths. The `absolute` type expands environment variables (e.g. `%USERPROFILE%`) via `ExpandEnvironmentVariables`.

## Config File Convention

| File | Format | Committed | Purpose |
|------|--------|-----------|---------|
| `*_example.jsonc` | JSONC | Yes | Documented templates and community lookup table |
| `*.json` | JSON | No (gitignored) | Runtime config, generated by `GenerateConfigs.ps1` |
| `backup_store.json` | JSON | No | Persists user-chosen backup drive path |
| `backup_manifest.json` | JSON | No (in backup store) | Timestamps and inventory of last backup |
| `ignored_apps.json` | JSON | No | Permanently ignored apps during detection |
| `project_config.json` | JSON | No (gitignored) | Project-level settings and flags (log level, size limits, backup flags). Added v0.2.0. |

## Two-Phase App Lifecycle

```
Pre-reinstall:  BackupConfigs.ps1  -> snapshot config files to backup store
Post-reinstall: InstallApps.ps1    -> install software
                RestoreConfigs.ps1 -> copy config files back from backup store
```

## Detection Pipeline (planned -- v0.4.0+)

```
ScanApps.ps1
  Source 1: winget list  -> Name, Id, Source (winget/msstore/manual)
  Source 2: Start Menu   -> .lnk resolution via WScript.Shell COM
  Deduplication          -> prefer winget entry, enrich with Start Menu metadata
       |
       v
GenerateConfigs.ps1      -> user categorises: MainApps / AdditionalApps / Tools / Ignore
       |                    writes apps.json
       v
ScanConfigs.ps1
  Tier 1: Lookup table   -> app_configs_example.jsonc (fast path)
  Tier 2: Fuzzy AppData  -> name matching in %APPDATA%, %LOCALAPPDATA%, %PROGRAMDATA%
  Tier 3: Install dir    -> config file patterns in app install directory
       |
       v
User confirmation        -> accept/reject fuzzy matches
       |
       v
Write app_configs.json
```

## Three-Tier Config Discovery (planned -- v0.5.0)

**Tier 1: Community lookup table**
- Check `app_configs_example.jsonc` for a known mapping by app name
- If found, use directly -- no scanning needed
- This is the fast path; the more apps the community contributes, the fewer fuzzy matches

**Tier 2: Fuzzy AppData scan**
- Scan `%APPDATA%`, `%LOCALAPPDATA%`, `%PROGRAMDATA%` for folders matching the app name
- Matching: exact name > name contains > contained within
- Filter: skip folders > configurable size limit (default 100MB)
- Exclusions: Temp, Cache, CrashDumps, logs, node_modules, npm-cache, pip, pnpm
- Skip folders last modified > 2 years ago
- Present candidates to user for confirmation

**Tier 3: Install directory scan**
- Check the app's install directory for config files
- Look for: `*.json`, `*.ini`, `*.cfg`, `*.xml`, `*.db`, `*.sqlite`
- Skip `bin/`, `lib/` subdirectories
- Same size and exclusion filters as Tier 2

## Security Model

### Current (v0.1.x)

- **DPAPI**: Browser saved passwords and session cookies are encrypted with machine-specific keys. They cannot be restored across Windows reinstalls. Documented and warned at runtime.
- **UAC**: `setup.bat` requests elevation. Scripts that modify system-level settings (device name, dev mode) require admin rights.
- **No credential storage**: AnniWin11 never stores or transmits passwords, API keys, or session tokens.

### Planned (v0.9.0)

Full security hardening:

1. **No symlink following** -- check `(Get-Item $path).LinkType` before any operation
2. **Path traversal prevention** -- validate user-provided paths with `Resolve-Path`
3. **Private key detection** -- warn (never include) files matching `*.key`, `*.pem`, `*.pfx`, `*.p12`, `id_rsa*`, `id_ed25519*`, `*.ppk`
4. **No execution of discovered files** -- ScanApps and ScanConfigs read metadata only
5. **Audit logging** -- all skipped items during scanning logged at DEBUG level
6. **Size limits** -- check available space before backup, warn if < 20% free
7. **Backup integrity** -- SHA256 checksum per file in `backup_manifest.json`, verified on restore

## Path Token System

Used in `app_configs.json` to define where each app stores its config:

| Token | Resolves to |
|-------|-------------|
| `appdata` | `%APPDATA%` (typically `C:\Users\X\AppData\Roaming`) |
| `appdata_roaming` | Same as `appdata` |
| `localappdata` | `%LOCALAPPDATA%` (typically `C:\Users\X\AppData\Local`) |
| `programdata` | `%PROGRAMDATA%` (typically `C:\ProgramData`) |
| `absolute` | Path with environment variables expanded (e.g. `%USERPROFILE%\.gitconfig`) |

## Data Flow

### Pre-Reinstall (Backup)

```
User -> Main.ps1 -> BackupConfigs.ps1
                      |
                      +-> Read app_configs.json
                      +-> For each app: resolve path -> robocopy to backup store
                      +-> winget export -> backup store
                      +-> Write backup_manifest.json
```

### Post-Reinstall (Restore)

```
User -> Main.ps1 -> InstallApps.ps1
                      |
                      +-> Read apps.json
                      +-> For each app: install via winget/internet/local/terminal
                 -> RestoreConfigs.ps1
                      |
                      +-> Read app_configs.json + backup_manifest.json
                      +-> For each app: copy config files from backup store
                      +-> DPAPI warning for browser configs
```
